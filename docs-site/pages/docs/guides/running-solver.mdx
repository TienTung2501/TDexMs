# Running a Solver

import { Steps, Callout, Tabs } from 'nextra/components'

How to set up and operate a SolverNet solver node that processes swap intents.

<Callout type="warning">
Running a solver requires a **funded Cardano wallet** with enough ADA to cover transaction fees and collateral.
On preprod testnet, use the [Cardano Faucet](https://docs.cardano.org/cardano-testnets/tools/faucet/) to get tADA.
</Callout>

## Architecture

```
┌──────────────────────────────────────────────────────┐
│                  Solver Engine                        │
│                                                      │
│  ┌──────────────┐  ┌──────────────┐  ┌────────────┐ │
│  │ Intent       │  │ Route        │  │ Batch      │ │
│  │ Collector    │──▶│ Optimizer    │──▶│ Builder    │ │
│  │ (Blockfrost) │  │ (direct/hop) │  │ (max 15)   │ │
│  └──────────────┘  └──────────────┘  └────────────┘ │
│                                            │         │
│  ┌──────────────┐  ┌──────────────┐        ▼        │
│  │ TX           │◀─│ Netting      │  ┌────────────┐ │
│  │ Submitter    │  │ Engine       │◀─│ Settlement │ │
│  │ (FIFO queue) │  │ (offset)     │  │ Builder    │ │
│  └──────────────┘  └──────────────┘  └────────────┘ │
└──────────────────────────────────────────────────────┘
```

## Setup

<Steps>

### Generate Solver Wallet

Create a new Cardano wallet for the solver:

```bash
# The solver uses a seed phrase stored in SOLVER_SEED_PHRASE env var
# Generate a 24-word mnemonic using any CIP-1852 compatible tool

# Or use the backend's built-in generation:
pnpm exec tsx -e "
  import { generateSeedPhrase } from '@lucid-evolution/lucid';
  console.log(generateSeedPhrase());
"
```

<Callout type="error">
**Never** reuse your personal wallet as the solver wallet. The solver spends UTxOs automatically.
Store the seed phrase securely — it controls the solver's funds.
</Callout>

### Fund the Solver Wallet

The solver needs ADA for:

| Purpose | Amount |
|---------|--------|
| Transaction fees | ~0.3–0.8 ADA per settlement TX |
| Collateral | 5 ADA (locked for Plutus script execution) |
| Operating buffer | 50+ ADA recommended |

For testnet:
```bash
# Get solver address
pnpm exec tsx -e "
  import { Lucid, Blockfrost } from '@lucid-evolution/lucid';
  const lucid = await Lucid(
    new Blockfrost('https://cardano-preprod.blockfrost.io/api', process.env.BLOCKFROST_API_KEY),
    'Preprod'
  );
  lucid.selectWallet.fromSeed(process.env.SOLVER_SEED_PHRASE);
  console.log(await lucid.wallet().address());
"
```

Send tADA to this address from the faucet.

### Configure Environment

Add these solver-specific variables to your `.env`:

```env
# Solver wallet
SOLVER_SEED_PHRASE="word1 word2 ... word24"

# Solver behavior
SOLVER_INTERVAL=30000              # Run every 30 seconds
SOLVER_ENABLED=true                # Enable/disable solver
SOLVER_MAX_BATCH_SIZE=15           # Max intents per TX
SOLVER_MIN_PROFIT_LOVELACE=0       # Min profit to process (0 = altruistic)
SOLVER_PARTIAL_FILL_ENABLED=true   # Allow partial fills

# TX execution limits
SOLVER_MAX_TX_CPU=14000000000      # 14 billion CPU units
SOLVER_MAX_TX_MEM=10000000         # 10 million memory units

# Chain connection
BLOCKFROST_API_KEY=preprodXXXXXXXXX
CARDANO_NETWORK=Preprod
```

pnpm install

## Solver Pipeline

Each iteration runs this 6-stage pipeline:

### Stage 1: Collect Intents

`IntentCollector` queries Blockfrost for all UTxOs at the escrow validator address:

- Parses the 10-field `EscrowDatum` from each UTxO
- Filters expired intents (past deadline)
- Filters already-processing intents (prevents double-processing)
- Returns validated intent list

### Stage 2: Filter & Validate

SolverEngine filters:
- **Stale intents** — already being processed by another iteration
- **Expired intents** — past deadline
- **FILLING intents** — currently mid-settlement
- **Invalid datums** — malformed or unparseable

### Stage 3: Route Optimization

`RouteOptimizer` finds the best execution path for each intent:

- **Direct route**: Swap directly in the target pool (most common)
- **Multi-hop route**: ADA → TOKEN_A → TOKEN_B (via ADA as intermediate)
- Calculates expected output and price impact per route
- Falls back to **partial fill** (binary search via `calculateMaxIntentFill`) if full fill causes too much impact

### Stage 4: Batch Building

`BatchBuilder` groups intents by pool and enforces constraints:

| Constraint | Limit |
|-----------|-------|
| Max intents per TX | 15 |
| Max CPU units | 14 billion |
| Max memory units | 10 million |
| Max TX size | ~16 KB |

Oversized batches are split. Priority ordering: earliest expiry first (urgency), then FIFO.

### Stage 5: Netting

`NettingEngine.analyze()` optimizes each batch:

1. Separates intents by direction (AToB vs BToA)
2. Calculates gross flows in each direction
3. Converts to common denomination using spot price
4. Computes **net residual** = |flowAtoB - flowBtoA|
5. Only the residual hits the AMM pool

**Example:**
```
Intent 1: 100 ADA → TOKEN (buy)
Intent 2:  50 TOKEN → ADA (sell, equivalent to ~80 ADA)

Gross ADA→TOKEN: 100 ADA
Gross TOKEN→ADA:  80 ADA equivalent

Net residual: 20 ADA → TOKEN (only this hits AMM)
Netting improvement: 80% (saved 80 ADA of pool impact)
```

Output allocation is proportional:
$$output_i = \frac{intent_i.amount}{\sum amounts} \times totalOutput$$

### Stage 6: Settlement

The solver builds and submits the settlement TX:

1. Consumes all escrow UTxOs in the batch
2. Consumes the pool UTxO (for AMM swap)
3. Burns all intent tokens
4. Sends calculated outputs to each user
5. Creates new pool UTxO with updated reserves
6. Signs with solver wallet

The `TxSubmitter` uses a singleton FIFO queue shared across 3 bot types (solver, chain sync, order monitor) to prevent UTxO contention.

## Monitoring

### Health Check

```bash
curl http://localhost:3001/v1/health
# { "status": "ok", "solver": { "enabled": true, "lastRun": "..." } }
```

### Admin Dashboard

```bash
# Solver status
curl http://localhost:3001/v1/admin/solver/status \
  -H "Authorization: Bearer $ADMIN_TOKEN"

# Start/stop solver
curl -X POST http://localhost:3001/v1/admin/solver/start \
  -H "Authorization: Bearer $ADMIN_TOKEN"

curl -X POST http://localhost:3001/v1/admin/solver/stop \
  -H "Authorization: Bearer $ADMIN_TOKEN"
```

### Key Log Messages

| Log | Meaning |
|-----|---------|
| `Found N pending intents` | Intents ready for processing |
| `Netting analysis: X% offset` | Percentage of flow offset by counter-trades |
| `Settlement TX submitted` | Batch settled successfully |
| `TX confirmed in slot` | On-chain confirmation |
| `No profitable batches` | No intents to process or min profit not met |
| `Blockfrost API budget: N/50000` | Daily API calls remaining |

## Troubleshooting

| Issue | Cause | Solution |
|-------|-------|----------|
| `INSUFFICIENT_FUNDS` | Solver wallet low on ADA | Add more tADA to solver address |
| `COLLATERAL_NOT_SET` | No collateral UTxO | Send 5+ ADA to solver, restart |
| `BLOCKFROST_RATE_LIMIT` | API calls exhausted | Wait or use a dedicated API key |
| `TX_SUBMIT_FAILED` | UTxO already spent | Normal during concurrency — auto-retries |
| `SCRIPT_EXECUTION_FAILED` | Validator rejection | Check datum/redeemer construction |
| `Iteration timeout` | TX took too long | Increase timeout, check network |

## Profit Model

The solver can earn profit from the **spread** between netting offset and AMM execution:

```
User wants: 100 ADA → at least 4.95 TOKEN (minOutput)
AMM gives:  100 ADA → 5.10 TOKEN (actual output)

Surplus: 0.15 TOKEN (solver can keep or pass to user)
```

Currently, the solver passes all surplus to users. Set `SOLVER_MIN_PROFIT_LOVELACE` to require minimum profit per batch.

<Callout type="info">
In the future, solver competition will allow multiple solvers to compete for intent batches, driving better execution for users.
</Callout>
