# Advanced Orders

import { Steps, Callout, Tabs } from 'nextra/components'

SolverNet supports limit orders, DCA (Dollar-Cost Averaging), and stop-loss orders — all enforced on-chain.

## Order Types

| Type | Description | Execution |
|------|-------------|-----------|
| **Limit** | Execute when price reaches target | Partial fills allowed, repeating |
| **DCA** | Buy/sell fixed amount at intervals | Time-based, `amount_per_interval` |
| **Stop-Loss** | Convert entire position at trigger | Full budget, single execution |

All orders are locked on-chain in the **Order Validator** UTxO with an intent token as thread token.

## Limit Orders

Execute a swap only when the market price meets your target.

### How It Works

```
┌──────────────┐          ┌───────────────────┐
│ User creates │─────────▶│   Order UTxO      │
│ limit order  │          │ OrderDatum {       │
└──────────────┘          │   type: LIMIT,     │
                          │   budget: 100 ADA, │
                          │   targetPrice,     │
                          │   filledAmount: 0  │
                          │ }                  │
                          └───────────────────┘
                                    │
                          Solver checks price
                                    │
                          ┌─────────▼─────────┐
                          │ Price meets limit? │
                          │ meets_limit_price  │
                          └─────────┬─────────┘
                             Yes    │    No
                              ▼     │     ▼
                          Execute   │  Wait
                          partial   │
                          fill      │
```

### Price Condition

The solver checks the on-chain AMM price satisfies your limit:

$$\frac{outputAmount}{inputAmount} \geq \frac{minOutput}{budget}$$

This is checked using integer arithmetic in the validator:

```
outputAmount × budget ≥ minOutput × inputAmount
```

### Creating a Limit Order

```json
POST /v1/orders
{
  "type": "LIMIT",
  "creator": "addr_test1...",
  "inputAsset": { "policyId": "", "assetName": "" },
  "outputAsset": { "policyId": "abc...", "assetName": "TOKEN" },
  "budget": "100000000",
  "minOutput": "5200000",
  "deadline": 1720000000,
  "senderAddress": "addr_test1...",
  "changeAddress": "addr_test1..."
}
```

### Partial Fills

Limit orders support partial execution:

- If the pool can only absorb part of your order at the target price, the solver fills that portion
- A new Order UTxO is created with updated `filledAmount` and reduced budget
- Continues until fully filled or deadline
- Same 10% minimum threshold as intents

## DCA Orders

Automatically buy/sell a fixed amount at regular intervals.

### How It Works

```
Budget: 300 ADA
Interval: 1 hour (3600 seconds)
Amount per interval: 100 ADA

Hour 0: Swap 100 ADA → TOKEN  ✓
Hour 1: Swap 100 ADA → TOKEN  ✓
Hour 2: Swap 100 ADA → TOKEN  ✓ (budget exhausted)
```

### Time Enforcement

The Order validator enforces interval timing:

```
min_dca_interval = 180 seconds (3 minutes)

Validator checks:
  tx_valid_from ≥ last_fill_time + interval
```

This uses Cardano's validity interval to enforce timing — the solver cannot submit a DCA fill transaction that claims to be before the interval has elapsed.

### Creating a DCA Order

```json
POST /v1/orders
{
  "type": "DCA",
  "creator": "addr_test1...",
  "inputAsset": { "policyId": "", "assetName": "" },
  "outputAsset": { "policyId": "abc...", "assetName": "TOKEN" },
  "budget": "300000000",
  "amountPerInterval": "100000000",
  "interval": 3600,
  "deadline": 1720086400,
  "senderAddress": "addr_test1...",
  "changeAddress": "addr_test1..."
}
```

### DCA Lifecycle

```
CREATED → ACTIVE → PARTIALLY_FILLED → ... → FILLED
                                          → EXPIRED (reclaim)
                                          → CANCELLED (cancel)
```

Each execution:
1. Fills `amountPerInterval` from the remaining budget
2. Sends output tokens to the creator
3. Creates continuing UTxO with updated datum (`filledAmount`, `budget - amountPerInterval`)
4. `fillCount` increments

## Stop-Loss Orders

Convert your entire position when price drops below a threshold.

### How It Works

```
You hold: 1,000 TOKEN
Stop price: 1 TOKEN = 18 ADA (current: 20 ADA)

If price drops to 18 ADA or below:
  → Sell all 1,000 TOKEN for ADA
  → Receive at least minOutput ADA
```

### Stop-Loss vs Limit

| Feature | Limit | Stop-Loss |
|---------|-------|-----------|
| Trigger | Price rises to target | Price drops to trigger |
| Execution | Partial fills | Full budget, single execution |
| Purpose | Buy low / Sell high | Protect against downside |
| Continuation | Yes (repeating) | No (one-shot) |

### Creating a Stop-Loss

```json
POST /v1/orders
{
  "type": "STOP_LOSS",
  "creator": "addr_test1...",
  "inputAsset": { "policyId": "abc...", "assetName": "TOKEN" },
  "outputAsset": { "policyId": "", "assetName": "" },
  "budget": "1000000000",
  "minOutput": "18000000000",
  "deadline": 1720086400,
  "senderAddress": "addr_test1...",
  "changeAddress": "addr_test1..."
}
```

## Managing Orders

### View Orders

- **UI**: Portfolio → Open Orders tab
- **API**: `GET /v1/orders?creator=addr_test1...`
- **API**: `GET /v1/portfolio/open-orders?wallet_address=addr_test1...`

### Cancel an Order

Cancel returns your remaining budget:

```
DELETE /v1/orders/:orderId
```

Or via portfolio:

```json
POST /v1/portfolio/build-action
{
  "wallet_address": "addr_test1...",
  "utxo_ref": "txhash#index",
  "action_type": "cancel"
}
```

Cancellation requires your signature (creator pubkey hash) and burns the intent token.

### Reclaim After Deadline

If an order expires without being fully filled:

```json
POST /v1/portfolio/build-action
{
  "wallet_address": "addr_test1...",
  "utxo_ref": "txhash#index",
  "action_type": "reclaim"
}
```

<Callout type="warning">
Reclaim is only available **after** the deadline. The on-chain validator enforces `tx_valid_from > deadline`.
</Callout>

## Order Status Flow

```
CREATED ─────────▶ SUBMITTED ─────────▶ ACTIVE
                                          │
                    ┌─────────────────────┤
                    │                     │
              CANCELLED             PARTIALLY_FILLED
              (user cancel)               │
                                    ┌─────┤
                                    │     │
                                 FILLED  EXPIRED
                              (complete)  (deadline)
                                          │
                                       RECLAIMED
                                    (user reclaim)
```

## On-Chain Guarantees

All order types share these validator-enforced guarantees:

| Guarantee | Enforcement |
|-----------|-------------|
| Price protection | `meets_limit_price` check on output vs budget ratio |
| Timing (DCA) | Validity interval ≥ last fill + interval |
| Ownership | Only creator can cancel (signature check) |
| Deadline | Reclaim only after deadline, fills only before |
| Token continuity | Intent token thread token pattern |
| Anti-replay | One-shot intent token burned on completion |
