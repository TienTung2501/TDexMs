## Intent Lifecycle

The core innovation of SolverNet DEX is the **intent-based trading model**. Users submit trade intentions (intents) rather than executing trades directly.

### Intent Flow

1. **User creates intent** вҶ’ Backend builds escrow TX вҶ’ User signs with wallet
2. **Intent goes ACTIVE** вҶ’ Solver detects it via chain sync
3. **Solver matches** вҶ’ Routes through AMM pool вҶ’ Builds settlement TX
4. **Settlement confirms** вҶ’ Intent marked FILLED вҶ’ User receives output tokens

## Background Services

Six background services run as cron jobs within the backend process:

| Service | Interval | Description |
|---------|----------|-------------|
| **SolverEngine** | 5s (`SOLVER_BATCH_WINDOW_MS`) | Collects active intents, routes through pools, builds and submits settlement TXs |
| **ChainSync** | 120s (`CHAIN_SYNC_INTERVAL_MS`) | Polls Blockfrost for pool UTxO state, reconciles with database |
| **PriceAggregation** | 60s (`CHART_SNAPSHOT_INTERVAL_MS`) | Aggregates PriceTick records into Candle OHLCV data |
| **ReclaimKeeper** | 60s | Finds expired intents/orders and triggers on-chain reclaims |
| **OrderExecutor** | 60s | Checks DCA/Limit/StopLoss conditions, executes ripe orders |
| **PoolSnapshot** | 1 hour | Snapshots pool state вҶ’ PoolHistory, computes ProtocolStats |

### Startup Sequence

```typescript
// 1. Load environment & validate
const config = loadConfig();

// 2. Initialize infrastructure
const db = new PrismaClient();
const cache = new UpstashCache(config.redis);
const blockfrost = new BlockfrostProvider(config.blockfrost);

// 3. Initialize domain services
const lucid = await Lucid.new(blockfrost, config.network);
const solver = new SolverEngine(lucid, db, cache);

// 4. Start HTTP server
const app = createExpressApp(config, db, cache, solver);
app.listen(config.port);

// 5. Start background services
solver.start();
chainSync.start();
priceAggregation.start();
reclaimKeeper.start();
orderExecutor.start();
poolSnapshot.start();
```

## Security Model

### Smart Contract Security

- **Validator-enforced invariants** вҖ” All state transitions are verified by Plutus V3 scripts
- **Datum-based authorization** вҖ” Only the intent creator can cancel their intent
- **Pool NFT uniqueness** вҖ” Each pool has a unique NFT that prevents pool duplication
- **Settings NFT thread** вҖ” Protocol settings are controlled by a thread NFT

### API Security

- **Rate limiting** вҖ” Per-IP rate limiting with stricter limits on write operations
- **Input validation** вҖ” Zod schema validation on all request bodies and query parameters
- **CORS** вҖ” Configurable origin whitelist
- **Helmet** вҖ” Security headers (CSP, HSTS, etc.)
- **Admin auth** вҖ” Admin endpoints verify wallet address matches `ADMIN_ADDRESS`

### Operational Security

- **Solver seed phrase** вҖ” Stored as environment variable, never logged
- **Database** вҖ” Managed PostgreSQL with SSL (Supabase)
- **Cache** вҖ” Encrypted Redis connections (Upstash)
