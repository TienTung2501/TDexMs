## Liquidity Flow

### Add Liquidity (Deposit)

```
1. User selects pool & amounts
2. Frontend: POST /pools/:poolId/deposit
3. Backend: Calculate LP tokens to mint
   - lpTokens = min(amountA * totalLP / reserveA, amountB * totalLP / reserveB)
4. Backend: Build deposit TX with Lucid
   - Consume pool UTxO
   - Add user's tokens to pool reserves
   - Mint LP tokens (LP Token Policy)
   - Produce new pool UTxO with updated reserves
5. User signs TX → Submit to chain
6. ChainSync detects pool state change → Update DB
```

### Remove Liquidity (Withdraw)

```
1. User specifies LP tokens to burn
2. Frontend: POST /pools/:poolId/withdraw
3. Backend: Calculate proportional share
   - amountA = lpTokens * reserveA / totalLP
   - amountB = lpTokens * reserveB / totalLP
4. Backend: Build withdraw TX
   - Consume pool UTxO
   - Burn LP tokens (LP Token Policy)
   - Return proportional assets to user
   - Produce new pool UTxO with reduced reserves
5. User signs TX → Submit to chain
```

## Order Flow (Limit/DCA/Stop-Loss)

```
┌────────────────────────────────────────────────────────────┐
│ 1. User creates order → POST /orders                      │
│ 2. Backend builds escrow TX → User signs                  │
│ 3. Order goes ACTIVE on-chain                             │
│                                                            │
│ ┌──────────────── Monitoring Loop ──────────────────┐     │
│ │ OrderExecutorCron runs every 60s:                  │     │
│ │                                                    │     │
│ │ For each ACTIVE order:                             │     │
│ │   LIMIT:     Check if pool price >= target price   │     │
│ │   DCA:       Check if interval elapsed             │     │
│ │   STOP_LOSS: Check if pool price <= trigger price  │     │
│ │                                                    │     │
│ │ If condition met:                                  │     │
│ │   → Build execution TX against pool                │     │
│ │   → Submit settlement                              │     │
│ │   → Update order status (FILLED/PARTIALLY_FILLED)  │     │
│ └────────────────────────────────────────────────────┘     │
│                                                            │
│ ReclaimKeeperCron:                                        │
│   → If deadline passed → Mark EXPIRED                     │
│   → Build reclaim TX → Return funds to creator            │
└────────────────────────────────────────────────────────────┘
```

## Cron Job Data Flows

### ChainSync (every 120s)

```
Blockfrost API
  │ Query pool script UTxOs
  ▼
Parse UTxO datums (Pool datum)
  │ Extract reserves, fees, state
  ▼
Compare with DB state
  │ If changed:
  ▼
Update Pool record in DB
  │
  ▼
Emit poolUpdate via WebSocket
  │
  ▼
Create PriceTick record
```

### PriceAggregation (every 60s)

```
PriceTick records (new since last run)
  │
  ▼
Group by poolId + interval
  │
  ▼
For each group:
  ├── Find/create Candle record
  ├── Update OHLCV values:
  │   open  = first tick price
  │   high  = max(existing high, tick prices)
  │   low   = min(existing low, tick prices)
  │   close = last tick price
  │   volume += sum(tick volumes)
  │   txCount += count(ticks)
  └── Save Candle
```

### PoolSnapshot (every 1 hour)

```
All ACTIVE pools
  │
  ▼
For each pool:
  └── Create PoolHistory record
      (reserveA, reserveB, tvlAda, volume, fees, price)
  │
  ▼
Compute ProtocolStats
  (aggregate TVL, volume, fees, pool count, etc.)
  │
  ▼
Save ProtocolStats record
```

## Error Recovery

<Callout type="info">
All TX-building operations are idempotent. If a TX fails to submit, the system can safely retry without double-spending.
</Callout>

| Failure | Recovery |
|---------|----------|
| TX submission fails | Retry with fresh UTxO set (max 3 retries) |
| Solver TX fails | Intent remains ACTIVE, retried next batch |
| ChainSync fails | Stale data until next successful sync |
| DB write fails | TX still on-chain; ChainSync reconciles |
| Redis unavailable | Fallback to DB queries (slower) |
