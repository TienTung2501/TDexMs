# Minting Policies

import { Callout } from 'nextra/components'

Three minting policies control token creation and destruction in SolverNet.

## Intent Token Policy

**File**: `validators/intent_token_policy.ak` (91 lines)
**Parameters**: None (unparameterized)
**Purpose**: One-shot NFT that identifies and tracks each swap intent / order.

### MintIntentToken

Creates a unique token bound to a specific UTxO:

```
Checks:
1. One-shot pattern: a specific UTxO is consumed in the TX
2. Token name = blake2b_256(txId ++ outputIndex) вҖ” derived from consumed UTxO
3. Exactly 1 token minted with that name
4. Guarantees globally unique token (UTxO can only be consumed once)
```

```aiken
// Token name derivation
fn derive_token_name(utxo: OutputReference) -> AssetName {
  blake2b_256(
    serialise(utxo.transaction_id) |> bytearray.concat(
      serialise(utxo.output_index)
    )
  )
}
```

<Callout>
The intent token policy has **no parameters**, unlike what older docs may suggest. It derives the escrow validator hash at runtime from the datum, not from script parameterization.
</Callout>

### BurnIntentToken

Destroys intent tokens during settlement or cancellation:

```
Checks:
1. Negative quantity in mint field (burning)
2. Allows multiple burns per TX вҖ” enables batch settlement
   where the solver burns tokens for all settled intents in one TX
```

This is critical for efficiency: a single settlement TX can burn 15 intent tokens simultaneously.

## Pool NFT Policy

**File**: `validators/pool_nft_policy.ak` (117 lines)
**Parameters**: `factory_validator_hash: ScriptHash`, `admin_vkh: VerificationKeyHash`
**Purpose**: Thread token that ensures pool UTxO continuity.

### MintPoolNFT

Creates the unique NFT that lives in a pool UTxO:

```
Checks:
1. One-shot pattern (consumed UTxO derives token name)
2. Exactly 1 token minted
3. Factory validator invoked in the same TX
   (ensures pool creation goes through proper validation)
```

The Pool NFT serves as a **thread token**:
- Every pool TX must prove the NFT continues from input to output
- Prevents attackers from creating fake pool UTxOs
- Token name is unique per pool (derived from creation UTxO)

### BurnPoolNFT

Destroys the NFT when closing a pool:

```
Checks:
1. Admin signature (admin_vkh)
2. Only admin can burn вҶ’ only admin can close pools
```

## LP Token Policy

**File**: `validators/lp_token_policy.ak` (104 lines)
**Parameters**: `pool_validator_hash: ScriptHash`, `factory_validator_hash: ScriptHash`
**Purpose**: Controls LP token minting and burning via forwarding.

### Forwarding Mint Pattern

The LP Token policy does **not** contain AMM math. Instead, it delegates validation:

```
For any mint/burn of LP tokens:
1. Check: Pool validator script hash appears in TX inputs or withdrawals
   OR Factory validator script hash appears in TX inputs or withdrawals
2. If yes вҶ’ LP token policy succeeds (Pool/Factory already validated the amounts)
3. If no вҶ’ LP token policy fails (unauthorized mint)
```

This ensures:
- **Deposits**: Pool validator verifies proportional deposit and correct LP amount
- **Withdrawals**: Pool validator verifies LP burned matches proportional withdrawal
- **Pool creation**: Factory validator verifies initial LP = вҲҡ(amountA Г— amountB) - minimum_liquidity
- LP tokens cannot be minted or burned outside of these validated operations

### LP Token Naming

Each pool has its own LP token with a unique asset name (derived from the pool creation UTxO). The policy ID is shared across all pools, but the asset name is unique per pool.

```
LP Token = (lp_token_policy_id, pool_specific_asset_name)
```

## Token Summary

| Token | Type | Parameters | Mint Rule | Burn Rule |
|-------|------|------------|-----------|-----------|
| Intent Token | One-shot NFT | None | UTxO consumed, blake2b name, exactly 1 | Any negative amount (batch burns) |
| Pool NFT | Thread Token | factory_hash, admin_vkh | UTxO consumed, factory invoked | Admin signature |
| LP Token | Fungible | pool_hash, factory_hash | Pool or Factory validator invoked | Pool validator invoked |
