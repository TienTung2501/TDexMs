# Validators

import { Callout } from 'nextra/components'

Detailed reference for all 5 SolverNet validators.

## Escrow Validator

**File**: `validators/escrow_validator.ak` (299 lines)
**Parameters**: None (unparameterized)
**Purpose**: Holds user funds during a swap intent.

### Datum: EscrowDatum

```aiken
type EscrowDatum {
  creator: VerificationKeyHash,       // Owner who can cancel
  input_policy_id: PolicyId,          // Input token policy
  input_asset_name: AssetName,        // Input token name
  output_policy_id: PolicyId,         // Desired output token policy
  output_asset_name: AssetName,       // Desired output token name
  min_output: Int,                    // Minimum acceptable output
  deadline: POSIXTime,                // Expiry timestamp
  escrow_token: AssetName,            // Intent token name (thread)
  input_amount: Int,                  // Original input amount
  max_partial_fills: Int,             // Max allowed partial fills (default: 5)
  fill_count: Int,                    // Current fill count (starts at 0)
  remaining_input: Int,               // Remaining input to be filled
}
```

<Callout>
The datum has **12 fields**, not 6 as in simplified docs. Fields like `fill_count`, `remaining_input`, and `max_partial_fills` enable partial fill functionality.
</Callout>

### Redeemers

#### Cancel
Owner reclaims funds before deadline.

```
Checks:
1. Creator signature present (check_signer)
2. Intent token burned (check_burn_one)
3. Funds returned to creator address
```

#### Fill
Solver settles the intent (fully or partially).

```
Checks:
1. Before deadline (check_before_deadline)
2. Output ≥ proportional minOutput:
   output ≥ (fill_amount × min_output) / input_amount
3. Complete fill:
   - Intent token burned
   - Output sent to creator via check_payment_output_secure
4. Partial fill:
   - fill_amount ≥ 10% of remaining_input
   - fill_count < max_partial_fills
   - Continuing UTxO with updated datum:
     - remaining_input -= fill_amount
     - fill_count += 1
   - Continuing UTxO has InlineDatum (anti-double-satisfaction)
   - Intent token continues to new UTxO
5. Anti-double-satisfaction:
   - check_payment_output_secure verifies InlineDatum contains
     the escrow_token name, preventing solver from using one
     output to satisfy two different intents
```

#### Reclaim
Owner reclaims after deadline expiry.

```
Checks:
1. After deadline (check_after_deadline: tx_valid_from > deadline)
2. Intent token burned
3. Funds returned to creator
```

## Pool Validator

**File**: `validators/pool_validator.ak` (459 lines)
**Parameters**: `admin_vkh: VerificationKeyHash`
**Purpose**: Constant-product AMM pool with fee accrual.

### Datum: PoolDatum

```aiken
type PoolDatum {
  asset_a: AssetClass,             // First token (canonically ordered)
  asset_b: AssetClass,             // Second token
  reserve_a: Int,                  // Active reserve of token A
  reserve_b: Int,                  // Active reserve of token B
  total_lp_tokens: Int,            // Total outstanding LP tokens
  fee_numerator: Int,              // Trading fee (e.g., 30 = 0.30%)
  protocol_fee_a: Int,             // Accumulated protocol fees (token A)
  protocol_fee_b: Int,             // Accumulated protocol fees (token B)
  pool_nft: AssetClass,            // Thread token identifying this pool
  last_root_k: Int,                // √(reserveA × reserveB) pre-swap
}
```

### Redeemers

#### Swap
Execute a token swap against the pool.

```
Active reserves = physical_value - protocol_fees

Checks:
1. Pool NFT continuity (thread token stays in pool UTxO)
2. Swap output = calculate_swap_output(input, reserves, fee)
3. Protocol fee accrued: fee × protocol_fee_share (1/6)
4. Constant product maintained:
   √(newReserveA × newReserveB) ≥ last_root_k
5. Datum updated: new reserves, fees, last_root_k
6. Physical value matches reserves + protocol_fees
```

**Swap formula** (integer arithmetic):
```
fee = input_amount × fee_numerator / fee_denominator
input_after_fee = input_amount - fee
output = (reserve_out × input_after_fee) / (reserve_in + input_after_fee)
```

#### Deposit
Add proportional liquidity.

```
Checks:
1. Pool NFT continuity
2. Deposit is proportional (within tolerance)
3. LP tokens minted:
   min(amountA × totalLP / reserveA, amountB × totalLP / reserveB)
4. LP token minting verified (forwarding to LP Token policy)
5. Datum updated: new reserves, totalLpTokens
```

#### Withdraw
Remove liquidity by burning LP tokens.

```
Checks:
1. Pool NFT continuity
2. LP tokens burned
3. Output proportional:
   outputA = lpBurned × reserveA / totalLP
   outputB = lpBurned × reserveB / totalLP
4. Datum updated
```

#### CollectFees
Admin collects accumulated protocol fees.

```
Checks:
1. Admin signature (admin_vkh)
2. Fees reset to zero in datum
3. Physical value reduced by collected fee amount
4. Reserves unchanged
```

#### ClosePool
Admin permanently closes the pool.

```
Checks:
1. Admin signature
2. Pool NFT burned
3. All remaining funds withdrawable
```

## Order Validator

**File**: `validators/order_validator.ak` (371 lines)
**Parameters**: `intent_token_policy_id: PolicyId`
**Purpose**: Handles limit, DCA, and stop-loss orders.

### Datum: OrderDatum

```aiken
type OrderDatum {
  creator: VerificationKeyHash,
  input_asset: AssetClass,
  output_asset: AssetClass,
  budget: Int,                     // Total input budget
  min_output: Int,                 // Minimum acceptable output
  deadline: POSIXTime,
  order_type: OrderType,           // LIMIT | DCA | STOP_LOSS
  order_params: OrderParams,       // Type-specific parameters
  filled_amount: Int,              // Total filled so far
  fill_count: Int,
  escrow_token: AssetName,         // Intent token (thread)
}

type OrderType {
  Limit
  DCA
  StopLoss
}

type OrderParams {
  amount_per_interval: Int,        // DCA: amount per execution
  interval: Int,                   // DCA: seconds between fills
  last_fill_time: Int,             // DCA: timestamp of last fill
}
```

### Redeemers

#### CancelOrder
Creator cancels and reclaims budget.

```
1. Creator signature
2. Intent token burned
3. Remaining budget returned
```

#### ExecuteOrder
Solver fills the order (dispatches by type).

**Limit Order:**
```
1. Before deadline
2. meets_limit_price: output × budget ≥ min_output × input
3. Partial fill → continuing UTxO with updated filled_amount
4. Full fill → intent token burned
```

**DCA Order:**
```
1. Before deadline
2. Fill amount = min(amount_per_interval, remaining_budget)
3. Timing check: tx_valid_from ≥ last_fill_time + interval
4. Continuing UTxO with updated last_fill_time, filled_amount
5. Full budget consumed → intent token burned
```

**Stop-Loss Order:**
```
1. Before deadline
2. Full budget converted (no partial fills)
3. Output ≥ min_output
4. Intent token burned
```

#### ReclaimOrder
After deadline, creator reclaims remaining budget.

```
1. After deadline (tx_valid_from > deadline)
2. Intent token burned
3. Remaining budget returned
```

## Factory Validator

**File**: `validators/factory_validator.ak` (212 lines)
**Parameters**: `pool_validator_hash: ScriptHash`
**Purpose**: Creates new pools with proper initialization.

### Redeemers

#### CreatePool

```
Checks:
1. Assets canonically ordered (asset_a < asset_b by policy then name)
2. Pool NFT minted (exactly 1)
3. LP tokens minted (√(amountA × amountB) - minimum_liquidity)
4. Pool UTxO created at pool_validator address with correct datum:
   - reserve_a = amountA
   - reserve_b = amountB
   - total_lp_tokens = minted LP amount
   - fee_numerator within bounds
   - protocol_fees = 0
   - last_root_k = √(amountA × amountB)
5. Pool NFT in the pool UTxO value
```

#### UpdateSettings

```
Checks:
1. Admin signature
2. Settings datum version incremented
```

## Settings Validator

**File**: `validators/settings_validator.ak` (98 lines)
**Parameters**: `settings_nft: AssetClass`
**Purpose**: Stores and controls protocol-wide configuration.

### Datum: SettingsDatum

```aiken
type SettingsDatum {
  admin: ScriptHash,               // Admin script hash (multi-sig pattern)
  fee_numerator: Int,              // Default trading fee
  protocol_fee_share: Int,         // Protocol's share (1/N)
  min_liquidity: Int,              // Minimum locked LP
  version: Int,                    // Config version (must increment)
}
```

### Redeemers

#### UpdateProtocolSettings

```
Checks:
1. Admin authorization via withdrawal check (script-hash admin)
2. Settings NFT continuity
3. Version incremented
4. Fee within bounds (min_fee_numerator..max_fee_numerator)
5. min_liquidity ≥ minimum_liquidity constant
```

<Callout type="info">
The settings validator uses a **script-hash-based admin** pattern. Instead of checking a simple VKH signature, it checks that the admin script (which can be a multi-sig) appears in the transaction's withdrawals. This enables flexible admin governance.
</Callout>
